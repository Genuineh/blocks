# blocks 抽象语言白皮书

> 状态说明（2026-03-09）：
> 本文描述的是 `BCL` 的目标形态与长期设计方向，不等同于“当前仓库已经完整实现”的状态。
> 当前仓库对 `BCL` 的正式落地边界，以 `docs/prds/BCL_MOC_ASSIST_PLAN.md` 与 `docs/specs/BCL_MOC_MVP_SPEC.md` 为准。
> 在当前 MVP 中，`BCL` 被严格约束为 `moc` 描述与校验的辅助层，`moc.yaml` 仍然是运行权威描述。

## 摘要

当 `block` 成为一种稳定、可验证、可评估的基础积木后，AI 的核心工作将不再是直接书写大量底层实现代码，而是围绕这些积木进行组织、编排、约束和交付。此时，AI 对 `block` 的使用本身，就会成为一种需要被审查、被验证、被编译的“上层代码”。

这意味着，`blocks` 体系不只需要组件规范和开发方法，还需要一门专门为 AI 设计的抽象语言，以及一套对应的编译器。该语言的目标，是让 AI 使用稳定的 `block` 描述完整产品的构成方式；该编译器的目标，是在真正生成运行产物之前，先检查 AI 对 `block` 的组织是否符合规则、是否存在错误、是否可以被验证，以及是否能够被装配为一个可运行的完整系统。

本文提出一套面向 `blocks` 的 AI 抽象语言设想，暂命名为 `BCL`，即 `Blocks Composition Language`。`BCL` 不是用来替代现有编程语言，而是作为一种更高层的、面向 AI 的编排语言：它将稳定的 `block` 作为基本语义单元，将完整产品的构建过程表达为一段可审查、可分析、可编译的组合代码。

## 1. 为什么需要一门 blocks 语言

### 1.1 当 block 稳定后，AI 的工作重心发生变化

在 `blocks` 模型中，底层能力不再直接暴露为难以控制的框架细节，而是被封装为一组稳定的可用积木。当这些积木足够稳定之后，AI 的主要任务就变成了：

- 选择哪些 `block`
- 如何连接它们
- 如何传递输入与输出
- 如何定义运行流程
- 如何设置约束、错误处理与验证

这类工作本质上已经是“编码行为”，只是编码对象不再是函数、类和底层库，而是稳定的能力单元。

### 1.2 如果没有语言，AI 的组织结果仍然不可控

即便底层 `block` 稳定，如果 AI 仍然通过自然语言、临时脚本或随意结构去组织它们，仍然会出现以下问题：

- 组织方式缺乏统一表达，难以审查。
- `block` 使用是否符合契约，无法被系统性检查。
- 输入输出连接是否正确，难以自动发现。
- 整体系统是否可运行，只能在运行时被动暴露。
- 每次 AI 的组织结果难以复现、复用和比较。

因此，仅有稳定的 `block` 还不够，必须有一层正式的语言，把 AI 的组织行为转化为显式、结构化、可验证的源代码。

### 1.3 需要把 AI 的“组织决策”变成可审查源码

在传统软件工程中，函数、类和模块组成了需要审查的代码。到了 `blocks` 体系中，AI 主要写的将不再是这些低层代码，而是“如何使用 `block` 构建产品”的高层代码。

所以，`BCL` 的核心价值是：

- 让 AI 写出来的不是模糊方案，而是正式源码。
- 让人类审查的是 `block` 的组织逻辑，而不是大量机械细节。
- 让编译器在运行前发现结构性错误，而不是把问题留到上线之后。

## 2. BCL 的定位

`BCL` 是一门面向 AI 的高层抽象语言，用于描述如何使用 `blocks` 构建一个完整产品。

它的定位是：

- 它是 `blocks` 体系的编排语言，不是底层实现语言。
- 它以 `block` 为基本语义单元，而不是以函数或类为基本语义单元。
- 它的源代码表达的是“产品构造逻辑”，不是“每个底层步骤的手工实现”。
- 它的编译结果是由多个 `block` 组装而成的、可运行、可验证的产品产物。

`BCL` 可以被看作是 AI 时代的软件“上层源码语言”。

## 3. 设计目标

`BCL` 应满足以下设计目标。

### 3.1 面向 AI 生成

语言必须足够规则化、低歧义、结构稳定，使 AI 能够稳定地产生合法代码，而不是高度依赖隐式约定。

### 3.2 面向人类审查

语言必须具备良好的可读性。人类工程师应能快速看懂：

- 这个产品用了哪些 `block`
- 它们如何连接
- 它们承担什么职责
- 可能在哪些地方失败

### 3.3 面向编译器分析

语言设计必须让编译器能够在编译期做结构检查、契约检查、类型检查、依赖检查和运行性检查，而不是只在运行期才发现问题。

### 3.4 面向完整产品输出

语言不能只停留在“描述流程”。它必须能够编译为一个可启动、可运行、可验证的完整产物，例如服务、任务流、接口程序或自动化系统。

### 3.5 面向稳定复用

同一份 `BCL` 源码应可重复编译、可复现构建、可对比版本变更。它必须是正式工程资产，而不是一次性指令。

## 4. 核心思想

`BCL` 基于以下核心思想。

### 4.1 block 是原子能力

在 `BCL` 中，`block` 是最小能力单元。语言不直接关心某个 `block` 内部如何实现，而关心：

- 它是否存在
- 它是否稳定
- 它的契约是什么
- 它如何与其他 `block` 协作

### 4.2 产品是可执行的 block 图

一个完整产品，在 `BCL` 中可以被理解为一张可执行图：

- 节点是 `block`
- 边是数据、事件或控制流
- 图上的约束描述运行条件、质量要求和失败处理方式

因此，`BCL` 的本质不是描述算法细节，而是描述可执行的能力图。

### 4.3 AI 写的是组装代码

在 `BCL` 中，AI 的主要工作是：

- 引用 `block`
- 声明产品入口
- 定义流程和分支
- 绑定输入与输出
- 声明策略和检查规则

这使 AI 的产出变成了一种新的、比传统源码更高层的工程资产。

## 5. BCL 的基本语言模型

`BCL` 建议至少由以下几个核心构造组成。

### 5.1 `product`

`product` 是顶层产物定义，表示最终要编译成什么完整系统。它定义：

- 产品名称
- 目标类型
- 入口
- 对外暴露接口
- 全局策略

### 5.2 `use`

`use` 用于声明当前源码依赖哪些 `block`，以及使用哪些版本约束。编译器据此解析依赖与兼容性。

### 5.3 `input` 与 `output`

用于定义产品级输入与输出边界，明确整个系统对外接受什么、返回什么。

### 5.4 `flow`

`flow` 用于定义主要执行流程，是 `BCL` 的核心。它描述：

- 哪些 `block` 被调用
- 调用顺序
- 条件分支
- 并行与串行关系
- 中间结果传递

### 5.5 `bind`

`bind` 用于显式绑定一个 `block` 的输入来源与输出去向，避免隐式传值。

### 5.6 `guard`

`guard` 用于声明运行前或运行中的规则约束，例如前置条件、权限要求、资源限制或兼容性要求。

### 5.7 `verify`

`verify` 用于声明编译期或运行期必须满足的验证规则，使最终产物不只是能运行，而且具备可证明的正确性基础。

### 5.8 `recover`

`recover` 用于定义当某个 `block` 失败时的回退、重试、替代或终止策略。

## 6. 基本语法提案

以下是一个最小 `BCL` 示例。

```bcl
product content_pipeline {
  type service
  runtime http

  use block input-parser@^1.0
  use block text-normalizer@^1.0
  use block summary-generator@^2.1
  use block result-validator@^1.2

  input {
    request.text: string
  }

  output {
    response.summary: string
  }

  flow main {
    step parse = input-parser
    step normalize = text-normalizer
    step summarize = summary-generator
    step validate = result-validator

    bind request.text -> parse.raw_text
    bind parse.cleaned_text -> normalize.raw_text
    bind normalize.normalized_text -> summarize.source_text
    bind summarize.summary -> validate.generated_text
    bind validate.accepted_text -> response.summary

    guard validate.status == "ok"
    verify main is runnable
  }
}
```

这个例子表达了几个关键点：

- 源码直接声明产品，而不是声明底层实现细节。
- 所有能力依赖通过 `use block` 显式引入。
- 所有数据流通过 `bind` 显式声明。
- `guard` 和 `verify` 将运行约束和可验证性纳入源码本身。

## 7. 语义规则

仅有语法还不够，`BCL` 必须有一套稳定语义规则。

### 7.1 block 引用语义

当 `use block x@y` 被声明时，编译器必须能解析：

- 该 `block` 是否存在
- 版本是否满足约束
- 状态是否允许被正式使用
- 契约是否完整

`draft` 或不完整的 `block` 不应进入正式编译路径。

### 7.2 绑定语义

每个 `bind a -> b` 都必须满足：

- 源端存在
- 目标端存在
- 源类型可赋给目标类型
- 目标前置条件不被破坏
- 不会形成非法循环依赖

### 7.3 流程语义

`flow` 的语义应保证：

- 流程存在确定入口
- 关键路径在逻辑上可达
- 无未处理的失败分支
- 无无法终止的死循环
- 所有必要输出都能被生产

### 7.4 验证语义

`verify` 不只是注释，而是可执行编译规则。任何不满足的 `verify` 条件都应导致编译失败或至少产生阻断级诊断。

## 8. BCL 编译器的职责

`BCL` 编译器不是简单的代码转换工具，它是 `blocks` 体系中的规则执行者与产品装配器。

说明：
本节描述的是长期目标形态。当前仓库已经完成 `BCL` 的 MVP 核心链路：`lexer/parser/IR/sema/plan/emit(parity)`，且只面向 `moc` 辅助产物，不直接生成运行时代码或部署产物。尚未完成的是仓库级 gate、warn/error rollout 与回滚开关，这些属于 MVP 之后的运营硬化阶段。

它至少要完成以下职责：

- 解析 `BCL` 源码
- 检查语法合法性
- 解析 `block` 依赖
- 加载 `block` 契约
- 校验输入输出匹配
- 检查流程可运行性
- 生成装配计划
- 输出可执行产物
- 生成验证报告

编译器的目标不是“尽量把东西拼起来”，而是“在规则可证明的前提下组装出可运行产品”。

## 9. 编译器架构提案

建议将 `BCL` 编译器划分为八个阶段。

### 9.1 词法与语法分析

将 `BCL` 源码解析为抽象语法树。此阶段负责发现：

- 关键字错误
- 结构不完整
- 标识符非法
- 语法块不闭合

### 9.2 语义建模

将语法树转换为语言语义模型，建立：

- 产品定义模型
- 流程模型
- 绑定关系模型
- 约束模型

### 9.3 block 解析与注册表查询

从 `blocks` 注册表或本地目录中加载被引用的 `block`：

- 读取 `block` 契约
- 检查状态与版本
- 建立依赖图

这是 `BCL` 编译器与 `blocks` 基础设施最关键的连接点。

说明：
“版本解析”是长期目标的一部分。当前 MVP 尚未引入 block 版本解析器或 lockfile，实际落地以“按本地快照 block id 解析”为准。

### 9.4 契约与类型检查

编译器应检查：

- 输入输出字段是否存在
- 类型是否兼容
- 必填项是否被满足
- 前置条件是否被破坏
- 副作用是否违反策略

这一阶段是 AI 自查能力的核心来源之一。

### 9.5 流程可运行性分析

编译器应检查：

- 是否有孤立节点
- 是否有无源输入
- 是否有未消费的关键输出
- 是否存在死路或非法环
- 失败分支是否被处理

这一步的目的，是在编译期提前发现“看起来合法，但实际上跑不起来”的系统。

### 9.6 装配计划生成

在通过前述检查后，编译器生成一份装配计划，描述：

- 各 `block` 的实例化顺序
- 数据与控制流连接关系
- 运行时需要的适配层
- 启动入口与生命周期管理

装配计划可以被视为中间表示，也可以被用于调试和审计。

### 9.7 代码生成与产物构建

编译器根据装配计划生成最终产物。产物可能包括：

- 运行时入口代码
- `block` 调度代码
- 配置文件
- 依赖清单
- 部署清单
- 验证脚本

这里生成的不是抽象描述，而是可启动的完整工程。

说明：
这同样是长期目标，不是当前仓库的 MVP 边界。当前 MVP 只允许生成 `moc.yaml`、规划产物和 parity/诊断结果，不生成 runtime/deploy 代码。

### 9.8 编译后验证

编译器应在构建后执行基础验证，例如：

- 启动检查
- 健康检查
- 关键路径 smoke test
- 必要配置完整性检查

如果产物无法通过这些最小验证，它不应被视为编译成功。

## 10. 编译器的错误模型

为了让 AI 能通过编译器发现自己的组织错误，`BCL` 编译器必须提供稳定、结构化的错误模型。

建议至少定义以下错误类别：

- `SyntaxError`：语法错误
- `UnknownBlockError`：引用了不存在的 `block`
- `VersionConflictError`：版本不兼容
- `ContractViolationError`：违反 `block` 契约
- `TypeMismatchError`：绑定类型不匹配
- `FlowUnreachableError`：流程不可达
- `MissingOutputError`：必要输出未被生成
- `RecoverPolicyMissingError`：失败路径缺少处理策略
- `RuntimeAssemblyError`：无法生成可运行装配
- `VerificationFailedError`：编译后验证失败

这些错误必须包含可定位信息，使 AI 和人类都能知道：

- 错误发生在哪一段源码
- 涉及哪个 `block`
- 违反了哪条规则
- 应优先修复什么

## 11. 中间表示与审查模型

`BCL` 编译器建议使用多层中间表示，以支持更强的分析和审查。

### 11.1 AST

用于表示源码语法结构。

### 11.2 Contract IR

将所有被引用 `block` 的契约统一加载后，形成标准化契约中间层，用于兼容性与规则分析。

### 11.3 Flow IR

将产品逻辑表示为一张显式的可执行图，便于发现孤立节点、错误依赖和运行问题。

### 11.4 Assembly IR

表示最终装配计划，描述产物如何落地为具体运行系统。

这些中间层有两个作用：

- 为编译器提供更强的检查能力。
- 为人类审查和 AI 自我诊断提供清晰的可视化对象。

## 12. 编译产物

`BCL` 的编译结果不应只是“生成一段代码”，而应是一个完整的可运行产品包。

一个标准编译产物建议至少包含：

- 入口程序
- 已解析的 `block` 依赖清单
- 调度与连接代码
- 运行配置
- 验证报告
- 构建元数据

换句话说，`BCL` 不是生成“片段”，而是生成“产品”。

说明：
上述描述适用于白皮书的理想终态。当前仓库的已批准 MVP 不直接编译出完整运行产品，而是先收口到 `moc` 辅助描述、语义校验与规范化生成。

## 13. BCL 对 AI 的意义

`BCL` 的提出，将改变 AI 在软件生产中的工作形态。

### 13.1 AI 从写底层代码转向写结构化编排代码

当底层 `block` 足够稳定时，AI 不再需要在每次任务中重复构造大量低层逻辑，而是直接产出更高层的结构代码。

### 13.2 AI 可以被编译器即时审查

AI 每次写出的 `BCL` 源码，都可以立即被编译器检查：

- 是否引用了合法 `block`
- 是否连接正确
- 是否违反契约
- 是否能被组装为完整产品

这使 AI 能在生成阶段获得接近“静态类型系统”的约束反馈。

### 13.3 AI 的产出成为正式工程资产

`BCL` 代码可以被审查、版本管理、对比变更、回归验证。它不是一次性 prompt 结果，而是正式的软件工程资产。

## 14. BCL 与传统语言的关系

`BCL` 并不替代 `Python`、`TypeScript`、`Go`、`Rust` 等传统语言。

它与传统语言的关系是：

- 传统语言负责实现 `block` 的内部能力。
- `BCL` 负责组织这些稳定能力形成完整产品。

因此，`BCL` 是一种更高层的编排抽象，其地位更接近“产品构造语言”，而不是通用编程语言。

## 15. 约束与边界

为了保持 `BCL` 的稳定性，需要明确它的边界。

### 15.1 不负责细粒度算法表达

复杂算法细节仍然应该在 `block` 内部实现，而不是在 `BCL` 中展开。

### 15.2 不鼓励把临时脚本伪装为 block 组合

若某个能力本身不稳定、不完整、不可验证，就不应被包装后直接进入 `BCL` 编译路径。

### 15.3 不绕过编译器规则

任何允许绕过契约检查、类型检查或验证检查的“逃逸机制”，都将破坏 `BCL` 的核心价值，应尽量避免。

## 16. 演进路线

`BCL` 的落地可以按以下阶段推进：

1. 定义最小语法子集，只支持 `product`、`use`、`flow`、`bind`。
2. 接入 `block` 契约解析，实现基础类型检查和依赖检查。
3. 生成最小可运行装配产物。
4. 增加 `guard`、`verify`、`recover` 等高级能力。
5. 引入可视化 `Flow IR` 和更强的编译诊断。
6. 建立标准运行时和部署模型。

这样可以先让语言可用，再逐步增强其表达力和工程能力。

相对于本文的理想形态，当前仓库仍有以下明确缺口：

## 17. 结论

当 `block` 成为稳定积木之后，AI 的主要创造行为将不再是反复书写底层实现，而是对这些积木进行高层组织。为了让这种组织行为可控、可验证、可审查、可编译，`blocks` 体系必须拥有一门专门为 AI 设计的抽象语言，以及一套能够执行规则、检查错误、生成产物的编译器。

`BCL` 的意义正在于此：它把 AI 对 `block` 的组织从“临时决策”提升为“正式源码”，把系统交付从“运行后才知道是否可用”提升为“编译时尽可能证明其正确性”，并把最终结果从“若干零散代码片段”提升为“由稳定能力组装而成的完整产品”。

这将使 AI 软件生产进入一个新的阶段：底层能力由 `block` 保证稳定，上层产品由 `BCL` 保证可审查，而编译器则成为连接这两者的质量控制中枢。
